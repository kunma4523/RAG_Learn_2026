# 为什么需要RAG？

深入分析RAG技术的必要性和应用价值。

## 1. 大模型的固有局限

### 1.1 知识截止

```
LLM知识 = 训练数据 知识截止日期
                    │
                    ▼
        ┌─────────────────────┐
        │  GPT-4: 2023-09    │
        │  Claude: 2023-08   │
        │  Qwen: 2023-12     │
        └─────────────────────┘
                    │
                    ▼
        无法回答之后的事件
```

### 1.2 幻觉问题

LLM可能生成看似合理但实际错误的内容：

```python
# 用户问: 2024年ACL最佳论文是什么?
# LLM可能编造:
# "根据我的知识，2024年ACL最佳论文是..."
# 实际上可能是幻觉
```

### 1.3 长尾知识

```python
# 通用问题 - LLM表现好
"什么是人工智能?"  ✓

# 专业/冷门问题 - LLM可能出错
"XXX公司2024年Q3财报具体数据?"  ✗
```

## 2. RAG的核心价值

### 2.1 知识实时性

```
RAG可以实时接入:
├── 最新新闻
├── 实时数据
├── 产品目录
├── 用户文档
└── 企业知识库
```

### 2.2 事实可验证性

```python
# RAG可以提供来源
answer = {
    "text": "RAG通过检索增强生成...",
    "sources": [
        {"doc": "doc_001.txt", "score": 0.95},
        {"doc": "doc_002.txt", "score": 0.87}
    ]
}
```

### 2.3 领域适应性

```
通用LLM
    │
    │ + RAG + 专业知识库
    ▼
领域专家
    │
    ├── 法律LLM ──▶ 接入法律条文库
    ├── 医疗LLM ──▶ 接入医学文献
    └── 金融LLM ──▶ 接入财报/研报
```

## 3. RAG vs 其他方案对比

| 方案 | 优点 | 缺点 | 适用场景 |
|------|------|------|----------|
| 纯Prompt | 简单 | 知识有限 | 通用问答 |
| 模型微调 | 适应任务 | 成本高，需数据 | 特定格式/风格 |
| RAG | 知识新，可追溯 | 依赖检索质量 | 知识密集型 |
| RAG+微调 | 最优 | 复杂 | 生产级应用 |

## 4. 什么时候用RAG？

```
需要RAG的场景:
├── 知识库频繁更新
├── 需要来源引用
├── 专业领域知识
├── 私有/内部数据
└── 长文档理解

不需要RAG的场景:
├── 通用知识问答
├── 创意/写作任务
├── 简单格式化任务
└── 风格转换任务
```
